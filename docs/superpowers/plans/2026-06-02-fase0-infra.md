# Fase 0 — Infra de Métricas (Datasources + Read-only) — Plano de Implementação

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Entregar a infraestrutura compartilhada que habilita as Fases 1–3 dos dashboards por aplicação: (a) descobrir e documentar a conexão Postgres de prod do nossalista; (b) confirmar e formalizar como o Grafana provisiona dashboards e datasources; (c) criar um usuário PostgreSQL **read-only** (`grafana_ro`) em cada banco de prod (nossalista e nossagrana) com SELECT-only; (d) provisionar 2 datasources PostgreSQL no Grafana via Helm values apontando para os Services in-cluster cross-namespace; (e) aplicar via `helm upgrade` e verificar; (f) documentar tudo no README do monitoring.

**Architecture:** Stack `kube-prometheus-stack` (Helm release `kps`, namespace `monitoring`, chart `kube-prometheus-stack-84.5.0`, Grafana app `v0.90.1` — Grafana real é a versão do subchart). Grafana em `grafana.leoferolive.com.br` (Ingress Traefik), backend Postgres externo. Datasources são provisionados por **arquivo de provisioning** gerado pelo chart: o bloco `grafana.datasources` (default Prometheus/Alertmanager) e `grafana.additionalDataSources` (nossos PG) são renderizados num Secret/ConfigMap que o sidecar de datasources monta. Dashboards são provisionados por **sidecar** (`grafana.sidecar.dashboards`, label `grafana_dashboard=1`, `searchNamespace: ALL`) lendo ConfigMaps — padrão já em uso pelo `chat-api-dashboard`. Os 2 novos datasources apontam para Services in-cluster cross-namespace por FQDN: `postgres.database.svc.cluster.local:5432` (nossalista) e `nossagrana-postgres.nossagrana.svc.cluster.local:5432` (nossagrana). Credenciais read-only via `secureJsonData.password` referenciando variáveis de ambiente injetadas por Secrets no pod do Grafana (mesmo mecanismo do `GF_DATABASE_PASSWORD` já existente).

**Tech Stack:** Kubernetes (K3s single-node, Raspberry Pi homelab `leo-ubuntu`), Helm v3, `kubectl` (acesso direto ao cluster), PostgreSQL 16, Grafana (subchart do kube-prometheus-stack). Tudo versionado no repo `chat-api-monitoring`.

## Questões abertas / riscos

- **RESOLVIDO (host nossalista):** era a questão bloqueante da spec; é o Postgres compartilhado `postgres.database.svc.cluster.local:5432`, db `nossalista`. Mesma instância do backend do Grafana → conectividade in-cluster trivial.
- **Divergência de nomes nossagrana:** a spec diz db/secret `nossagrana`; o cluster real usa db **`nossagrana_prod`** e user **`nossagrana_prod`**. O plano usa os valores reais. Reconfirmar no Task 1.
- **Recursos do RPi:** nossagrana-postgres roda no Raspberry Pi com recursos limitados. O usuário read-only é só SELECT; o risco real é nas queries das Fases 1–2 (manter leves/indexadas, refresh moderado). Para Fase 0 o impacto é nulo (só DDL de role + grants).
- **Cross-namespace:** datasources apontam para Services em namespaces diferentes (`database`, `nossagrana`). K3s não tem NetworkPolicy por padrão → tráfego in-cluster liberado. Verificar conectividade no Task 5 (teste de query no datasource).
- **`additionalDataSources` vs editar o ConfigMap:** NÃO editar `kps-grafana-datasource` à mão (o chart sobrescreve no próximo upgrade). Usar `grafana.additionalDataSources` no values.yaml (chave padrão do subchart Grafana, merge no provisioning).
- **SELECT em tabelas futuras:** `GRANT SELECT ON ALL TABLES` cobre só tabelas existentes no momento. Adicionar `ALTER DEFAULT PRIVILEGES` para tabelas criadas depois por migrations.

## Task 3 — Criar usuário PostgreSQL read-only (`grafana_ro`) em cada banco de prod

- [ ] Criar `k8s/monitoring/sql/grafana_ro_nossalista.sql` com o conteúdo EXATO (idempotente, role + grants + default privileges; sem INSERT/UPDATE/DELETE):
  ```sql
  -- grafana_ro para o database nossalista (Postgres compartilhado postgres.database).
  -- Idempotente: re-rodar atualiza a senha. Senha via psql -v pw=...
  -- READ-ONLY: somente CONNECT + USAGE + SELECT. Nenhuma permissão de escrita.

  DO $$
  BEGIN
    IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'grafana_ro') THEN
      EXECUTE format('CREATE ROLE grafana_ro LOGIN PASSWORD %L', :'pw');
    ELSE
      EXECUTE format('ALTER ROLE grafana_ro WITH LOGIN PASSWORD %L', :'pw');
    END IF;
  END $$;

  GRANT CONNECT ON DATABASE nossalista TO grafana_ro;
  GRANT USAGE ON SCHEMA public TO grafana_ro;
  GRANT SELECT ON ALL TABLES IN SCHEMA public TO grafana_ro;
  -- Tabelas criadas no futuro (migrations) ficam legíveis automaticamente:
  ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO grafana_ro;

  -- Garantia explícita de que NÃO há escrita (revoga heranças de PUBLIC, se houver):
  REVOKE INSERT, UPDATE, DELETE, TRUNCATE ON ALL TABLES IN SCHEMA public FROM grafana_ro;
  ```
  OUTPUT ESPERADO: arquivo criado.

- [ ] Criar `k8s/monitoring/sql/grafana_ro_nossagrana.sql` com o conteúdo EXATO (idêntico, trocando o database para `nossagrana_prod`):

  ```markdown
  ## Datasources Postgres read-only (métricas de negócio)

  Os dashboards de negócio (nossalista, nossagrana) consultam o Postgres de
  cada app via um usuário **read-only** (`grafana_ro`, SELECT-only). Um usuário
  por banco; a senha vai num Secret do namespace `monitoring`, injetada no pod
  do Grafana via `envFromSecrets` e referenciada por `secureJsonData.password`
  nos `additionalDataSources` do `values.yaml`.

  **Conexões de prod (confirmadas no cluster):**

  | App        | Service (FQDN)                                          | Porta | Database         | Datasource uid  |
  |------------|---------------------------------------------------------|-------|------------------|-----------------|
  | nossalista | `postgres.database.svc.cluster.local`                   | 5432  | `nossalista`     | `nossalista-pg` |
  | nossagrana | `nossagrana-postgres.nossagrana.svc.cluster.local`      | 5432  | `nossagrana_prod`| `nossagrana-pg` |

  > nossalista usa o **Postgres compartilhado** do namespace `database` (mesma
  > instância do backend do Grafana). nossagrana tem Postgres dedicado no seu
  > namespace. O host do nossalista não está em manifesto — foi descoberto
  > decodificando `DATABASE_URL` do Secret `nossalista-secrets`.

  ### Criar o usuário read-only em cada banco

  Idempotente (re-rodar atualiza a senha). Os scripts DDL estão em
  `k8s/monitoring/sql/grafana_ro_*.sql` (só SELECT — sem INSERT/UPDATE/DELETE).
  A senha é passada via `psql -v pw=` para não vazar em `ps` nem no SQL.

  ```bash
  # --- nossalista (Postgres compartilhado, ns database) ---
  GRAFANA_RO_NL_PASS=$(openssl rand -hex 16)
  PG_POD=$(kubectl get pods -n database --no-headers | head -1 | awk '{print $1}')
  PG_ROOT_USER=$(kubectl -n database get secret postgres-secret -o jsonpath='{.data.postgres-user}' | base64 -d)
  PG_ROOT_PASS=$(kubectl -n database get secret postgres-secret -o jsonpath='{.data.postgres-password}' | base64 -d)
  kubectl exec -n database "$PG_POD" -i -- env PGPASSWORD="$PG_ROOT_PASS" \
    psql -U "$PG_ROOT_USER" -d nossalista -v ON_ERROR_STOP=1 -v pw="$GRAFANA_RO_NL_PASS" \
    < k8s/monitoring/sql/grafana_ro_nossalista.sql
  kubectl create secret generic grafana-ds-nossalista -n monitoring \
    --from-literal=GF_DS_NOSSALISTA_PASSWORD="$GRAFANA_RO_NL_PASS" \
    --dry-run=client -o yaml | kubectl apply -f -

  # --- nossagrana (Postgres dedicado, ns nossagrana) ---
  GRAFANA_RO_NG_PASS=$(openssl rand -hex 16)
  NG_POD=$(kubectl get pods -n nossagrana --no-headers | grep nossagrana-postgres | head -1 | awk '{print $1}')
  NG_ROOT_USER=$(kubectl -n nossagrana get secret nossagrana-postgres-secrets -o jsonpath='{.data.POSTGRES_USER}' | base64 -d)
  NG_ROOT_PASS=$(kubectl -n nossagrana get secret nossagrana-postgres-secrets -o jsonpath='{.data.POSTGRES_PASSWORD}' | base64 -d)
  kubectl exec -n nossagrana "$NG_POD" -i -- env PGPASSWORD="$NG_ROOT_PASS" \
    psql -U "$NG_ROOT_USER" -d nossagrana_prod -v ON_ERROR_STOP=1 -v pw="$GRAFANA_RO_NG_PASS" \
    < k8s/monitoring/sql/grafana_ro_nossagrana.sql
  kubectl create secret generic grafana-ds-nossagrana -n monitoring \
    --from-literal=GF_DS_NOSSAGRANA_PASSWORD="$GRAFANA_RO_NG_PASS" \
    --dry-run=client -o yaml | kubectl apply -f -
  ```

  ### Provisionar os datasources

  Os 2 datasources estão em `grafana.additionalDataSources` no `values.yaml`,
  com a senha vinda das env vars dos Secrets acima (`grafana.envFromSecrets`).
  Aplicar com o `helm upgrade` padrão (ver seção "Upgrade") e verificar:

  ```bash

  GF_ADMIN_PASS=$(kubectl -n monitoring get secret grafana-admin -o jsonpath='{.data.admin-password}' | base64 -d)
  kubectl -n monitoring port-forward svc/kps-grafana 3000:80 &
  curl -s -u "$GF_ADMIN_USER:$GF_ADMIN_PASS" http://localhost:3000/api/datasources/uid/nossalista-pg/health
  curl -s -u "$GF_ADMIN_USER:$GF_ADMIN_PASS" http://localhost:3000/api/datasources/uid/nossagrana-pg/health
  # Esperado em cada: {"status":"OK", ...}
  ```

  > **Read-only de verdade:** `grafana_ro` não tem INSERT/UPDATE/DELETE. Teste
  > negativo: `psql -U grafana_ro ... -c "CREATE TABLE t(id int);"` deve dar
  > `permission denied for schema public`.
